# Sprint 3 Plan

## Title
Sprint 3 — Software Factory Architecture, Assistant Roles, and First Orchestrated Demo

## Goal
Build the first believable bridge between `neural_kernel` as a technically mature framework and the future startup direction as an AI-assisted software delivery company.

## Strategic Objective
Sprint 3 should establish:

1. a clear software factory architecture
2. a role-based assistant system
3. a first orchestrated demo scenario
4. a credible startup-facing positioning narrative

The goal is not to build a production platform yet.
The goal is to create the first structured and demoable software factory concept.

## Scope

### In Scope
- software factory architecture definition
- assistant role definition
- orchestrated demo design
- startup positioning definition
- artifact-based pipeline planning
- first demo deliverable specification

### Out of Scope
- broad autonomous code generation claims
- production deployment platform
- full commercial backend/frontend implementation
- support for arbitrary project types
- agent memory platform
- complex orchestration runtime
- full client delivery automation

### Demo Definition
A first orchestrated demo flow that takes:
- a customer request
- through assistant stages
- into structured artifacts

## Core Demo Direction

### Selected Demo Scenario
Internal knowledge assistant for a company.

### Why This Scenario
This scenario is selected because it is:
- easy to explain
- realistic
- useful for businesses
- narrow enough for a first orchestrated demo
- aligned with startup positioning

## First Demo Artifact Set
The first orchestrated demo should produce:

1. `project_brief.md`
2. `technical_spec.md`
3. `project_skeleton.md`
4. `review_report.md`
5. `delivery_summary.md`

These artifacts can initially be generated as structured markdown outputs.

## Assistant Pipeline
The initial assistant sequence is:

1. Intake Assistant
2. Solution Architect Assistant
3. Builder Assistant
4. QA / Review Assistant
5. Delivery Assistant

This order should remain fixed in Sprint 3 for clarity and simplicity.

## Execution Principles
- keep the system human-in-the-loop
- favor structured artifacts over vague agent output
- favor repeatability over “autonomy theater”
- keep demo scope narrow
- make business value easy to understand
- connect technical depth to commercial positioning

## Success Criteria

Sprint 3 is successful if all of the following are true:

- the software factory architecture is clearly documented
- assistant roles are clearly separated
- the first orchestrated demo is clearly specified
- the demo supports a believable startup story
- the project can be explained as more than just a framework
- the result feels focused rather than vague

## Priority Order

### Priority 1
Architecture clarity

### Priority 2
Assistant role clarity

### Priority 3
First orchestrated demo specification

### Priority 4
Startup positioning clarity

## Proposed Work Sequence

1. define architecture
2. define assistant roles
3. define first orchestrated demo
4. define startup positioning
5. create sprint backlog
6. define demo execution artifacts
7. review business-facing narrative

## Risks

### Risk 1 — Too broad positioning
Mitigation:
Keep the demo and startup story focused on one narrow use case.

### Risk 2 — Overclaiming autonomy
Mitigation:
Keep the system explicitly human-supervised and artifact-driven.

### Risk 3 — Architecture without demoability
Mitigation:
Tie every architecture document back to the first orchestrated demo.

### Risk 4 — Framework/business disconnect
Mitigation:
Make `neural_kernel` a strategic internal asset, not the only thing being sold.

## Expected End-of-Sprint State
By the end of Sprint 3, the project should have:

- a believable software factory architecture
- a role-based assistant pipeline
- a narrow business-facing demo scenario
- a clear startup positioning narrative
- a clear next step toward execution of the first orchestrated demo

## Summary
Sprint 3 is the transition sprint from framework maturity toward startup architecture.

It is the sprint where `neural_kernel` stops being only a mini deep learning framework and starts becoming part of a broader AI-assisted software delivery story.
